# AGENTS.md

Этот репозиторий предназначен для разработки максимально близкой по ощущению Godot-копии `ZERO Sievert`. Главный приоритет: не придумывать требования, не расползаться по архитектуре, не терять контекст после сжатия истории и не раздувать проект бессистемными правками.

## Главные правила

- Не додумывай факты о продукте, визуале, фичах или архитектуре. Если решение не вытекает прямо из текущих документов, кода или явного запроса пользователя, сначала спроси.
- При любой значимой неоднозначности задавай вопрос с `2-4` реальными вариантами ответа и кратко фиксируй рекомендуемый вариант.
- Не делай массовые скриптовые правки, широкие авто-рефакторы и codegen-проходы там, где можно исправить проблему точечно.
- Не добавляй runtime/editor addon'ы и внешние зависимости без явного выбора пользователя. Любое исключение сначала фиксируй в `docs/process/decision_log.md`.
- Не запускай большие реорганизации папок и модулей без отдельного запроса, даже если это кажется "чище".

## Token Efficiency Policy

- Сначала используй Serena, `rg`, точечный просмотр символов и локальные документы. Полное чтение файлов делай только если без этого нельзя принять решение.
- Избегай дублирования длинных контекстов в ответах. Ссылайся на уже созданные документы и логи.
- Не создавай лишние промежуточные файлы, если та же цель достигается обновлением существующих документов.
- Любой новый документ должен иметь конкретную долгосрочную роль. Если файл не станет источником истины, его не нужно создавать.

## Архитектурные guardrails

- Структура проекта: feature-first. Основные зоны: `features/raid`, `features/bunker`, `features/inventory`, `features/combat`, `features/traders`, `features/progression`, `ui`, `content`, `tools`, `runtime`.
- Статические данные описывай через typed `Resource`-модели. Правила и формулы держи в service/use-case классах. Сцены и контроллеры не должны становиться контейнером всей бизнес-логики.
- Runtime state не хранить строковыми флагами по UI и сценам. Держи отдельные модели save/session/run state.
- Autoload-ы разрешены только для кросс-фичевых сервисов. Перед добавлением нового singleton'а сначала докажи, что это действительно кросс-фичевая зависимость.
- Не допускать "тысячестрочных" скриптов. Если файл растет, смешивает UI, state и domain rules, разбивай по ответственности.
- Поддерживай typed GDScript и небольшие публичные API. Любой скрипт должен иметь одну понятную ответственность.

## Workflow

- После каждого meaningful task обновляй:
  - `docs/process/task_log.md`
  - нужные source-of-truth документы в `docs/`
  - Serena memory для overview/style/commands/completion или milestone state
- Любое архитектурное решение, смену курса, допущение по визуалу, зависимостям или scope фиксируй в `docs/process/decision_log.md`.
- Перед началом работы сверяйся с:
  - `docs/vision/product_north_star.md`
  - `docs/vision/visual_bible.md`
  - `docs/reference/zero_sievert_feature_matrix.md`
  - `docs/architecture/project_architecture.md`

## Локальные skills

- `godot-ask-first`: запрещает агенту додумывать требования и требует задавать уточняющие вопросы при неоднозначности.
- `godot-architecture-guardrails`: держит feature-first структуру, typed `Resource`-модели и минимальный autoload footprint.
- `context-sync`: напоминает обновлять документы и Serena memory после каждого meaningful task.
